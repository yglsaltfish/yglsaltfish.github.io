简单分析lower_bound在关联式容器中使用的性能问题

发表于 2023-12-14 分类于 C++
本文字数： 2.4k

前言

最开始是在写相关题目的时候碰到的这个问题，lower_bound或者upper_bound在关联式容器中存在两种用法，分别是

set<int> iSet;
lower_bound(iSet.begin(), iSet.end(), val);     // O(n)
iSet.lower_bound(val);                          // O(logn)

这两种用法会调用两种不同的实现方式来完成二分查找，接下来我从源码上来进行分析为什么会产生这种时间复杂度。

不同的lower_bound

简单写一段程序进行验证，然后单步调试查看分别调用的方法；

set<int> iSet;
void solve()
{
    lower_bound(iSet.begin(), iSet.end(), 1);
    iSet.lower_bound(1);
}

泛用的lower_bound

中间一些无关紧要的步骤略过，最终二分实现是如下：

template<typename _ForwardIterator, typename _Tp, typename _Compare>
   _GLIBCXX20_CONSTEXPR
   _ForwardIterator
   __lower_bound(_ForwardIterator __first, _ForwardIterator __last,
	  const _Tp& __val, _Compare __comp)
   {
     typedef typename iterator_traits<_ForwardIterator>::difference_type
_DistanceType;

     _DistanceType __len = std::distance(__first, __last);

     while (__len > 0)
{
  _DistanceType __half = __len >> 1;
  _ForwardIterator __middle = __first;
  std::advance(__middle, __half);
  if (__comp(__middle, __val))
    {
      __first = __middle;
      ++__first;
      __len = __len - __half - 1;
    }
  else
    __len = __half;
}
     return __first;
   }

这段代码中的while循环是标准的二分写法，时间复杂度是O(logn)，那么“罪魁祸首”大概率就是std::distance()函数了，单步进入查看实现：
 template<typename _InputIterator>
   inline _GLIBCXX14_CONSTEXPR
   typename iterator_traits<_InputIterator>::difference_type
   __distance(_InputIterator __first, _InputIterator __last,
              input_iterator_tag)
   {
     // concept requirements
     __glibcxx_function_requires(_InputIteratorConcept<_InputIterator>)

     typename iterator_traits<_InputIterator>::difference_type __n = 0;
     while (__first != __last)
{
  ++__first;
  ++__n;
}
     return __n;
   }

我们发现，这部分完全就是粗暴的while循环来查找lower_bound的范围长度，时间复杂度自然的就是O(n)了，所以总的时间复杂度就是O(logn + n)

特化的lower_bound

由于set的底层实现是红黑树，需要知道一些红黑树的基本特性，如红黑树是自平衡二叉查找树，左子树上所有节点的值均小于它的根节点的值，右子树上所有节点的值均大于它的根节点的值。跳过一些步骤，可以发现最终实现的源代码如下所示：

template<typename _Key, typename _Val, typename _KeyOfValue,
	   typename _Compare, typename _Alloc>
    typename _Rb_tree<_Key, _Val, _KeyOfValue,
		      _Compare, _Alloc>::iterator
    _Rb_tree<_Key, _Val, _KeyOfValue, _Compare, _Alloc>::
    _M_lower_bound(_Link_type __x, _Base_ptr __y,
		   const _Key& __k)
    {
      while (__x != 0)
	if (!_M_impl._M_key_compare(_S_key(__x), __k))
	  __y = __x, __x = _S_left(__x);
	else
	  __x = _S_right(__x);
      return iterator(__y);
    }

由于lower_bound要找的是第一个<=的值，所以一直往左子树找，找到一个比当前小的往右子树找，递归这个过程就可以了。

总结

一个简单的方法解决性能问题，同一个方法对于不同的容器，如果存在有容器有相对应的方法，那么优先使用容器对应的方法；这里lower_bound使用distance()方法也很合理，因为传入的是两个迭代器，并没有办法知道要寻找的区间长度，所以需要首先知道区间长度，以后有空可以写一篇迭代器相关的文章，:)。

秋招基本结束了，签了一家心仪的公司，之后开心玩耍了几周。准备秋招期间没时间读这本书，现在需要静下心来看书了。先说一下为什么选择看这本书吧，有一个比较经典的面试题是C++的编译阶段，基础回答就是 预处理，编译，链接，汇编。预处理就是处理一些宏、注释，编译涉及很多，语法分析、词法分析等，这个《编译原理》那本大黑书有提到，有时间的话我看完这本书之后去看，链接分为静态链接和动态链接，还有很多其他步骤，这中间做了什么事情我一直不是很清楚，这本书大部分讲的就是这个过程，汇编就是将汇编代码翻译成机器码。由于面试经历和个人兴趣，我准备去详细了解一下这个链接过程，准备看两遍，第一遍先囫囵吞枣，有个大致的概念；第二遍精读看细节。

1 总览

由于更习惯Linux系统，所以我用linux做例子。我们编译一个简单的C++源文件的时候，通常会使用g++ -Wall A.cpp -o a 简单命令来生成可执行文件，如果想要看更加细节的生成过程，可以使用--verbose命令。如果一个源代码文件需要使用其他文件的函数，可以使用g++ main.cpp A.cpp B.cpp -o a，然后我们就会得到一个可执行文件，可以使用readelf -e a 来查看这个文件的一些信息，在Section Headers中，我们可以看到很多段，有一些熟悉的比如.rodata ,.bss, .data段，这些在可执行文件中被称为section的东西，在执行过程中会根据读写权限不同分到不同的segment，这部分涉及到进程的内存分布，暂且不讲。从上述命令的结果中，我们可以知道有非常多的section，编译过程应该是逐步编译的，我们通过--verbose参数就可以发现是一共执行了三次编译过程，并且还执行了一些链接库函数的过程。接下来就是我要讲的部分，也就是静态链接的过程。
